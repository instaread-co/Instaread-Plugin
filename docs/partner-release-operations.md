# Partner Release Operations

Day-to-day reference for releasing partner plugin updates. Assumes the repo lives at `instaread-co/Instaread-Plugin` (post-2026-05-20 migration).

## Quick links

- Workflow: [.github/workflows/partner-builds.yml](../.github/workflows/partner-builds.yml)
- Core code: [core/instaread-core.php](../core/instaread-core.php)
- Telemetry endpoint: `https://player-api.instaread.co/api/plugin-telemetry`
- Webhook endpoint: `https://player-api.instaread.co/api/plugin-telemetry/github-webhook`

## Concept: how a partner release works

```
1. Edit partners/<id>/config.json   (per-partner settings, slugs, target selector)
   OR edit core/instaread-core.php  (shared logic affecting all partners)
2. Bump version in partners/<id>/config.json
3. Commit & push to main
4. Dispatch partner-builds.yml with partner_id + version
5. Workflow:
   - Generates plugin.json from template (overwriting any local edits)
   - Builds zip from core/ + partners/<id>/ + plugin-update-checker/
   - Patches plugin header to inject partner name + version via sed
   - Creates GitHub Release tagged <partner_id>-v<version>, uploads zip
   - Commits the regenerated plugin.json back to main
   - Fires webhook to player-api → notifies any telemetry-registered partner sites
6. Partner site:
   - Polls plugin.json via plugin-update-checker (every 12h by default)
   - OR receives webhook ping if previously registered → polls immediately
   - WP Plugin_Upgrader downloads zip from download_url
   - Installs new build, sends telemetry update event
```

## Single-partner release

```bash
# 1. Edit config.json if needed (e.g. add a selector, exclude a slug)
vim partners/<id>/config.json

# 2. Bump version in BOTH locations:
#    - partners/<id>/config.json  "version" field
#    - the workflow's `version` input (passed at dispatch time)
#    NOTE: plugin.json's version will be regenerated by the workflow, no manual edit needed

# 3. Commit
git add partners/<id>/config.json
git commit -m "fix(<id>): <what changed and why>"
git push origin main

# 4. Dispatch
gh workflow run partner-builds.yml -R instaread-co/Instaread-Plugin \
  -f partner_id=<id> -f version=4.5.X

# 5. Watch
sleep 4
RUN_ID=$(gh run list -R instaread-co/Instaread-Plugin --workflow partner-builds.yml --limit 1 --json databaseId -q '.[0].databaseId')
gh run watch -R instaread-co/Instaread-Plugin "$RUN_ID" --exit-status

# 6. Verify release
gh release view <id>-v4.5.X -R instaread-co/Instaread-Plugin

# 7. Verify webhook fired (sites_notified: N where N >= 1 if site is registered)
gh run view "$RUN_ID" -R instaread-co/Instaread-Plugin --log | grep sites_notified
```

## Fleetwide release (all partners)

**Use cases:**
- Core change that needs to propagate to every partner (URL migration, security fix, new feature)
- Repo-wide setting change (e.g. enable a new partner-config flag fleet-wide)

**MUST be sequential** — concurrent dispatches race on `git push` and ~80% fail. See [[fleetwide-release-dispatch]] memory.

### Sequential dispatcher

A working sequential dispatcher is preserved at `/tmp/dispatch_sequential.sh` after each fleetwide operation. Pattern:

```bash
#!/usr/bin/env bash
set -euo pipefail
VERSION="4.5.X"
REPO="instaread-co/Instaread-Plugin"

# Build partner list
ls partners/ > /tmp/partner_list.txt

while IFS= read -r partner; do
  [[ -z "$partner" ]] && continue
  echo "=== $partner ==="

  # Capture newest run id BEFORE dispatch so we can find the one we just fired
  before_id=$(gh run list -R "$REPO" --workflow partner-builds.yml --limit 1 \
              --json databaseId -q '.[0].databaseId')

  gh workflow run partner-builds.yml -R "$REPO" \
    -f partner_id="$partner" -f version="$VERSION"

  # Poll for our run to appear
  for i in 1 2 3 4 5 6 7 8; do
    sleep 2
    new_id=$(gh run list -R "$REPO" --workflow partner-builds.yml --limit 1 \
             --json databaseId -q '.[0].databaseId')
    [[ -n "$new_id" && "$new_id" != "$before_id" ]] && { run_id="$new_id"; break; }
  done

  # Watch to completion before next dispatch
  gh run watch -R "$REPO" "$run_id" --exit-status --interval 3 >/dev/null
  echo "  ✓ $partner"
done < /tmp/partner_list.txt
```

**Runtime:** ~25–30s per partner. 95 partners = ~45 min.

### Pre-flight: pilot 3 partners first

Before running across the fleet:

```bash
head -3 /tmp/partner_list.txt > /tmp/pilot.txt
# Run sequential dispatcher with /tmp/pilot.txt
# If 3/3 succeed, proceed with the rest
```

### Audit after fleetwide release

```bash
# Count v4.5.X releases on GitHub
gh release list -R instaread-co/Instaread-Plugin --limit 200 --json tagName | \
  python3 -c "
import json, sys
data = json.load(sys.stdin)
released = sum(1 for r in data if r['tagName'].endswith('-v4.5.X'))
print(f'{released} releases at v4.5.X')
"

# Find which partners are missing
python3 <<'EOF'
import json, subprocess, os, glob
out = subprocess.run(['gh','release','list','-R','instaread-co/Instaread-Plugin','--limit','200','--json','tagName'], capture_output=True, text=True)
released = {r['tagName'].rsplit('-v4.5.X',1)[0] for r in json.loads(out.stdout) if r['tagName'].endswith('-v4.5.X')}
all_partners = sorted(os.path.basename(os.path.dirname(p)) for p in glob.glob('partners/*/config.json'))
missing = [p for p in all_partners if p not in released]
print(f'Missing ({len(missing)}):', missing)
EOF
```

## Telemetry verification (did it actually install?)

```bash
# Did this partner's site install the new version?
curl -sS https://player-api.instaread.co/api/plugin-telemetry | \
  python3 -c "
import json, sys
data = json.load(sys.stdin)
partner = '<id>'
rows = [r for r in data if r.get('partner_id') == partner]
print(f'{partner}: {len(rows)} telemetry rows')
for r in sorted(rows, key=lambda x: x.get('ts',''))[-5:]:
    print(f'  {r[\"ts\"]}  {r[\"event\"]}  v{r.get(\"old_version\")} -> v{r[\"version\"]}')
"
```

## Verifying the player on a live page

```bash
URL="https://<partner-domain>/<some-article-slug>"

# Plugin version meta tag (server-side rendered)
curl -sS -L --max-time 30 "$URL?_t=$(date +%s)" -o /tmp/page.html
grep -oE '<meta[^>]*instaread-version[^>]*>' /tmp/page.html

# Player tag present?
grep -c '<instaread-player' /tmp/page.html

# Player slot context
python3 -c "
import re
with open('/tmp/page.html') as f: html=f.read()
for m in re.finditer(r'<div[^>]*class=\"instaread-player-slot\"[^>]*>.*?</div>', html, re.DOTALL):
    print(m.group())
    print('---')
"
```

## Inspecting a built zip locally

Sometimes you need to verify what's actually inside the deployed zip (e.g. confirming a URL migration or core patch landed):

```bash
TAG="<id>-v4.5.X"
mkdir -p /tmp/check && cd /tmp/check
gh release download "$TAG" -R instaread-co/Instaread-Plugin --pattern "${TAG}.zip"
unzip -q "${TAG}.zip"

# Verify version constant
grep "PLUGIN_VERSION = " */instaread-core.php

# Verify URLs (post-migration sanity check)
grep -n "suvarna-sumanth\|instaread-co" */instaread-core.php
```

## Common patterns

### "I need to add a slug to a partner's exclude list"

```bash
vim partners/<id>/config.json   # add to exclude_slugs array
# bump version in same file
git add partners/<id>/config.json
git commit -m "feat(<id>): exclude /<slug>/ from injection"
git push
gh workflow run partner-builds.yml -R instaread-co/Instaread-Plugin -f partner_id=<id> -f version=4.5.X
```

### "Partner's player is rendering in the wrong place"

Most placement issues are config — `target_selector` + `insert_position` in `injection_rules[0]`. See [docs/config-scenarios.md](config-scenarios.md) and [docs/partner-config-reference.md](partner-config-reference.md).

For WordPress sites where the `the_content` filter is used, target_selector should usually be empty (lets the plugin prepend directly to the article body). See [[wordpress_injection_rule]] in memory.

### "Partner says player vanished after update"

Could be:
1. **Pre-existing empty `instaread-player-slot` div** in their content (legacy from v2.8.x JS injection era) — handled by core guard since v4.5.10. Update them.
2. **Their theme/site builder injects a competing player** — config-side debugging.
3. **`force_disable_logs` is OFF and their cache caches the player markup** — check telemetry to confirm they actually upgraded.

See `docs/2026-05-incident-postmortem.md` for the moonandspoonandyum case study.

### "I need to roll back a partner"

Easiest: re-dispatch the workflow with an older version. The workflow regenerates everything from the partner config + core code at the time of dispatch.

If you need to roll back the CODE (not just version label), revert the commit in `core/` or `partners/<id>/`, push, then dispatch.

## Things to NEVER do

- ❌ Dispatch the workflow with parallel partners (race condition on `git push`)
- ❌ Edit `plugin.json` directly without running the workflow — the workflow overwrites it on every release
- ❌ Hardcode debug flags in `core/instaread-core.php` (`self::$debug = true`) — caused the 2026-05 outage
- ❌ Force-push to main (breaks the workflow's commit-back step)
- ❌ Delete the `suvarna-sumanth` GitHub account or create another `Instaread-Plugin` repo under it (breaks redirects for pre-v4.5.12 deployed sites)
